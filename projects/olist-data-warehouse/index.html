Project Overview

The Olist E-Commerce Data Warehouse is a fully automated, end-to-end data pipeline built on top of the publicly available Olist Brazilian E-Commerce dataset on Kaggle. The dataset contains approximately 100,000 orders, spanning customers, sellers, products, payments, reviews, and geolocation data across Brazil.

The project simulates a real-world data engineering workflow: raw CSV files arrive daily, are ingested into a landing zone, cleansed, modeled into an analytical warehouse, and surfaced to stakeholders via a Power BI dashboard — all triggered automatically by Apache Airflow.

End-to-End Scope This project covers every stage of the data engineering lifecycle: ingestion with Python, cleansing with T-SQL Stored Procedures, modeling as a Star Schema in SQL Server, orchestration with Airflow in Docker, and analytics with Power BI DAX.

Pipeline Architecture

The pipeline follows the Medallion Architecture — a three-layer pattern that progressively refines data quality from raw landing to business-ready analytics. Apache Airflow acts as the orchestrator for the entire flow.

🥉 Bronze Layer — Ingestion & Landing Zone

The Bronze layer is the raw landing zone. Its only job is to faithfully capture data exactly as it arrives from the source, adding audit metadata for traceability. No transformations, no business logic — just raw ingestion.

The Inbound → Archive workflow

The Python ingestion script (extract_to_bronze.py) watches a directory for new CSV files and manages a strict state machine for each file:

  • Inbound/ — files waiting to be processed
  • Processing/ — files currently being loaded (prevents double processing)
  • Archive/ — successfully loaded files
  • Error/ — failed files, isolated for inspection
Schema Drift Protection via dtype=str All CSV columns are read as raw strings using Pandas' dtype=str parameter. This prevents Pandas from inferring types that could crash the pipeline when the source system silently changes a column from integer to float (a common real-world hazard).
extract_to_bronze.py # Read ALL columns as strings to prevent Schema Drift crashes df = pd.read_csv( filepath, dtype=str, # ← no type assumptions keep_default_na=False ) # Append audit columns for every row df['_source_file_name'] = os.path.basename(filepath) df['_etl_load_datetime'] = datetime.utcnow() # Bulk-insert into SQL Server bronze schema df.to_sql( table_name, con=engine, schema='bronze', if_exists='append', index=False, chunksize=5000 )

🥈 Silver Layer — Cleanse & Conform

Nine T-SQL Stored Procedures handle the entire Bronze → Silver transformation. Each procedure handles one entity (orders, customers, products, etc.) and applies three techniques in sequence: type casting, deduplication, and incremental loading.

🔀

Safe Type Casting

TRY_CAST() converts raw strings to proper SQL types; invalid values become NULL instead of crashing the pipeline.

🔍

Deduplication

ROW_NUMBER() OVER(PARTITION BY ...) identifies and removes duplicate keys introduced by source systems.

🔄

Incremental UPSERT

SQL MERGE statement matches historical keys and UPDATEs changed rows or INSERTs new ones — no full reloads.

Deduplication with ROW_NUMBER()

Source systems sometimes emit duplicate keys for the same entity. Blindly loading these would corrupt analytics. The Silver layer filters them using a Window Function:

silver_customers.sql WITH deduplicated AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY customer_unique_id -- business key ORDER BY _etl_load_datetime DESC -- keep latest ) AS rn FROM bronze.olist_customers ) SELECT * FROM deduplicated WHERE rn = 1

Incremental Loading with SQL MERGE

Instead of truncating and reloading the entire Silver table on every pipeline run (expensive, and causes brief data unavailability), the MERGE statement performs surgical updates — only touching rows that changed:

silver UPSERT pattern (MERGE) MERGE silver.customers AS target USING ( SELECT * FROM deduplicated_bronze ) AS source ON target.customer_unique_id = source.customer_unique_id WHEN MATCHED AND (target.city != source.city OR ...) THEN UPDATE SET target.city = source.city, ... -- row changed → UPDATE WHEN NOT MATCHED BY TARGET THEN INSERT (customer_unique_id, city, state, ...) -- new row → INSERT VALUES (source.customer_unique_id, ...);

🥇 Gold Layer — Data Modeling

The Gold layer is where Data Modeling magic happens. Silver data is denormalized and restructured into a Star Schema optimized for analytical queries. Six Dimension tables and three Fact tables are populated by nine Stored Procedures that run sequentially (Dimensions first, then Facts — enforced by FK constraints).

Late-Arriving Facts / Missing Dimension Lookups When a Fact record references a Dimension that hasn't been loaded yet, the Foreign Key join would fail. The solution: pre-insert "Dummy Rows" into every Dimension with special Surrogate Keys — -1 for "Unknown" and -2 for "Not Yet Delivered". Every Fact MERGE uses COALESCE(lookup_sk, -1) to gracefully degrade instead of dropping rows. Power BI dashboards never show missing totals.

Role-Playing Dimensions

The DIM_DATE table is a centralized, statically generated date dimension. Rather than creating separate date tables for each time context, a single Date Dimension serves multiple roles within the same Fact table via aliased Foreign Keys:

  • purchase_date_sk → DIM_DATE (when was the order placed?)
  • delivered_date_sk → DIM_DATE (when did it arrive?)
  • approved_date_sk → DIM_DATE (when was payment approved?)

Power BI automatically handles the role-playing joins — the single dimension table is referenced multiple times, enabling synchronized cross-filtering across all three date contexts.

Solving Cartesian Explosion (Fan-out)

The Fan-out Problem When you join a 1-to-N table without pre-aggregating, each match creates N copies of the "1" side. If an order has 5 items, joining payments to items at the wrong grain creates 5 × payment_value instead of 1 — silently doubling (or worse) your financial metrics. This is catastrophic in a warehouse because BI tools would report inflated revenue.

The Olist dataset has this exact problem: one order can have multiple items (order_items) and multiple payment installments (order_payments). Joining them naively produces a Cartesian product.

The solution is pre-aggregation before joining — aggregate payments to order-level grain before building FACT_ORDER_ITEMS, and keep FACT_ORDER_PAYMENTS as a completely separate Fact table at payment-row grain:

gold_FACT_ORDER_ITEMS.sql — Fan-out prevention -- Step 1: Pre-aggregate payments to order grain BEFORE joining WITH payments_agg AS ( SELECT order_id, SUM(payment_value) AS total_payment, COUNT(payment_sequential) AS payment_count FROM silver.order_payments GROUP BY order_id -- ← collapse N rows to 1 per order ), -- Step 2: Deduplicate DIM_CUSTOMERS to ensure 1:1 join grain -- (customer_unique_id is the true key, not customer_id) unique_customers AS ( SELECT customer_id, customer_unique_id, ROW_NUMBER() OVER ( PARTITION BY customer_unique_id ORDER BY customer_id ) rn FROM silver.customers ) SELECT * FROM unique_customers WHERE rn = 1 -- Step 3: Build Fact at correct item grain using pre-aggregated dimensions INSERT INTO gold.FACT_ORDER_ITEMS SELECT oi.order_id, oi.order_item_id, -- natural composite key dp.product_sk, ds.seller_sk, dc.customer_sk, COALESCE(dd.date_sk, -1) purchase_date_sk, oi.price, oi.freight_value FROM silver.order_items oi JOIN silver.orders o ON oi.order_id = o.order_id JOIN unique_customers c ON o.customer_id = c.customer_id -- 1:1 now JOIN gold.DIM_CUSTOMERS dc ON c.customer_unique_id = dc.customer_unique_id ...

⏱ Orchestration with Apache Airflow

The entire pipeline is orchestrated by a Dockerized Apache Airflow instance. The DAG (olist_pipeline.py) runs on a daily schedule and manages three phases with different parallelism strategies.

🐍

Task 1 — Ingestion

PythonOperator triggers the Bronze ingestion script, processing all files in the Inbound directory.

⚡

Task 2 — Silver (Parallel)

All 9 Silver Stored Procedures run simultaneously inside a TaskGroup — independent tables, maximum parallelism.

🔗

Task 3 — Gold (Sequential)

Gold SPs run in strict order (Steps 1→9). Dimensions populate before Facts to satisfy FK constraints.

Docker + host.docker.internal Airflow runs inside Docker containers while SQL Server runs natively on the host Windows machine. The critical connection setting is Host: host.docker.internal in the Airflow MsSqlOperator connection, which routes from the container network out to the host machine's Port 1433 via TCP/IP.

Key Engineering Challenges Solved

1. Schema Drift in CSV Sources

Source CSV files occasionally change column types between runs (a numeric column arrives as text, or a date column gains milliseconds). Loading without protection crashes the pipeline. Solution: always ingest as dtype=str, then apply TRY_CAST() in Silver — graceful NULL instead of failure.

2. Duplicate Keys from Source Systems

The Olist dataset contains customers registered across multiple orders, with the same customer_unique_id appearing under different customer_id values. Joining on the wrong key explodes the grain. Solution: build DIM_CUSTOMERS around customer_unique_id (the true natural key) with ROW_NUMBER() deduplication in both Silver and the Gold dimension load.

3. Cartesian Explosion (Fan-out)

The most subtle and dangerous issue. Orders with multiple items joined to orders with multiple payments create a Cartesian product — multiplying financial metrics by N. Solution: pre-aggregate payments to order grain before joining, and keep FACT_ORDER_PAYMENTS as a completely separate Fact table at its natural payment-row grain.

4. Late-Arriving Facts

Orders that haven't been delivered yet have no delivered_date_sk to look up. A hard FK constraint would reject the entire order row. Solution: DIM_DATE pre-populates a dummy row with date_sk = -2 labeled "Not Yet Delivered". All Fact inserts use COALESCE(delivered_date_sk, -2) so undelivered orders are still captured and aggregated correctly in Power BI.
